# 2026-03-16

- Investigated disk usage on the Ubuntu server.
- Main disk pressure causes found:
  - systemd journal in `/var/log/journal` (~3.9G before cleanup)
  - `/var/lib/containerd` (~5.0G before cleanup)
  - `/var/lib/snapd` (~993M before cleanup)
  - old PM2 logs in `/root/.pm2` (~777M before cleanup)
- Found repeated error spam from `tvheadend` / `libav` and DVB USB kernel messages related to a MyGica T230A device.
- Found repeated SSH brute-force style attempts in logs from `172.16.23.1`.
- Disabled Snap services `snap.tvheadend.tvheadend.service` and `snap.tvheadend.oscam.service` and disabled their auto-start.
- Blacklisted DVB USB modules to prevent reloading after reboot:
  - `dvb_usb_dvbsky`
  - `dvb_usb_v2`
  - `m88ds3103`
- Verified tvheadend/DVB spam stopped after disable/deinit.
- Vacuumed systemd journal from ~3.9G down to ~552M.
- Cleaned PM2 logs from ~777M down to ~45M.
- Ran Docker/containerd cleanup (`docker system prune -a` / cache cleanup); `/var/lib/containerd` dropped from ~5.0G to ~2.8M.
- Cleared Snap cache; `/var/lib/snapd` dropped from ~993M to ~787M.
- Root filesystem free space improved from ~7.9G (87%) to ~18G (71%).
- Determined that the SSH brute-force source IP `172.16.23.1` is the host's default gateway and DHCP server on `ens18`, so the server sees SSH attacks as coming from the upstream gateway/router (or NAT/proxy in front of it), not from Docker.
- Likely next security investigation step: verify whether SSH port 22 is exposed externally through that gateway/NAT.
- Cleared current SSH auth log history by truncating `/var/log/auth.log` and deleting rotated `/var/log/auth.log.*` archives.
- Added `scripts/watch-ssh-bruteforce.sh` to watch for the next new brute-force line in `/var/log/auth.log`.
- Started a background subagent watcher (`ssh-bruteforce-watch`) with a 24-hour timeout to notify when a new brute-force entry appears.
